# ADR 0054: 학사일정 커넥터를 실제 소스(ssu.ac.kr)로 교정

- 상태: 승인
- 날짜: 2026-06-21
- 범위: 공개 도구 `get_academic_calendar`, `find_academic_calendar_events` / `RealAcademicCalendarConnector`

## 배경

prod의 `ssuai.connector.academic-calendar`는 오버라이드 없이 `application.yml` 기본값 `mock`으로 동작하고 있었다. 즉 공개 도구 `get_academic_calendar`/`find_academic_calendar_events`가 `MockAcademicCalendarConnector`의 **하드코딩된 가짜 2026 일정**(개강 2026-03-02, 기말 2026-06-15 등)을 내보내고 있었다. 크래시는 없지만 사용자·포트폴리오 데모에 **조용히 틀린 데이터**가 나가는 상태였다.

기존 `RealAcademicCalendarConnector`는 존재했고 주석에 "2026 레이아웃에 맞춰 셀렉터 확인함"이라고 적혀 있었지만, 실제로는 한 번도 라이브 페이지에 검증된 적이 없었다. 검증 결과 이 커넥터는 **세 군데가 동시에 틀려** 있었다.

1. **도메인**: `scatch.ssu.ac.kr/학사일정`을 긁었는데, 이 URL은 정상 User-Agent로도 **HTTP 404**(WordPress "페이지가 없습니다")다. scatch(SSU:catch)는 뉴스·홍보 포털이라 학사일정 페이지 자체가 없다(메뉴: 공지사항·뉴스센터·홍보물 등).
2. **파라미터**: 연도를 `?syear={year}`로 넘겼다.
3. **셀렉터**: `table.academic-calendar tbody tr` 등 실제 페이지에 존재하지 않는 추측성 셀렉터였다.

404 페이지에는 셀렉터를 "확인"할 수 없으므로, 주석의 "확인함"은 사실이 아니었다. 따라서 단순히 `real`로 켜면 404 → `ConnectorUnavailableException`이 되어 mock보다 더 나빠진다. 사용자 지시("go")는 "서버에서 실제 스크래핑이 되는지 검증한 뒤 real로 전환"이었다.

## 검토한 대안

### 대안 1: mock 유지 + "근사치" 라벨 또는 도구 비활성화
- 장점: 코드 변경 없음.
- 기각: 실제 데이터를 제공하지 못한다. 진짜 소스를 찾으면 해결되는 문제를 회피만 하는 것이고, 포트폴리오에서 "정보 도구가 가짜 데이터"라는 약점이 남는다.

### 대안 2: AJAX/JSON 내부 API 역공학
- 장점: 구조화 데이터를 HTML 파싱 없이 받을 수 있어 가장 견고했을 것.
- 기각: 실제 학사일정 페이지를 조사한 결과 **캘린더용 AJAX가 없다**. 페이지에서 유일하게 localize된 `admin-ajax`는 `search_number`(전화번호 검색) 플러그인용이고 일정과 무관했다. 일정 데이터는 서버 렌더링된 정적 HTML에 들어 있어 역공학할 JSON 엔드포인트 자체가 존재하지 않는다.

### 대안 3: 헤드리스 브라우저(Playwright 등)로 JS 렌더링
- 장점: 클라이언트 렌더링 페이지도 처리 가능.
- 기각: 이 페이지는 JS 렌더링이 아니다(정적 HTML에 일정이 이미 존재). 무거운 인프라를 정당화할 근거가 없다.

### 대안 4: 캘린더 이미지 OCR
- 장점: 이미지로만 게시된 경우의 마지막 수단.
- 기각: 콘텐츠 영역에 캘린더 이미지가 없다(`wp-content/uploads` img 0건). 불필요.

### 대안 5: 메인 사이트 정적 페이지 스크래핑 (채택)
- 실제 학사일정은 **메인 `ssu.ac.kr/학사/학사일정/`**에 있고, `?years={year}`로 연도를 선택한다(페이지의 prev/next 링크가 `?years=2025`/`?years=2027`을 사용). 서버 렌더링이라 Jsoup으로 바로 파싱되며, 월 블록마다 `id="calendarYYYYMM"`라는 견고한 앵커가 있다. 라이브 2025/2026/2027 페이지로 URL·파라미터·구조를 모두 실측 검증했다.
- 선택: 진짜 소스를 정확히 짚었고, 구조가 안정적이며, 검증 가능하다.

## 결정

`RealAcademicCalendarConnector`를 다음과 같이 재작성한다.

- **URL**: `{ssuai.academic-calendar.base-url:https://ssu.ac.kr}` + URL-encoded `/학사/학사일정/` + `?years={year}`. notice가 scatch base-url을 쓰는 것과 분리해 **전용 base-url 설정**을 신설했다(같은 base-url 재사용은 틀린 도메인 결합을 다시 만들기 때문).
- **파싱**: `div[id~=^calendar\d{6}$]` 월 블록 → 블록 안의 `ul.tb > li > div.row` → 날짜 셀(`div.text-primary`) + 제목 셀(날짜 셀이 아닌 나머지 자식). 날짜 셀 텍스트는 `"MM.DD (요일)"`(단일) 또는 `"MM.DD (요일) ~ MM.DD (요일)"`(기간)이다.

### 구현 레벨 선택 (근거 포함)

1. **연도는 블록 id, 월·일은 날짜 토큰에서**: 날짜 셀에는 연도가 없다(`MM.DD`). 연도는 `calendarYYYYMM` 블록 id가 권위 있는 앵커다. 반대로 월·일을 블록 id가 아니라 토큰에서 뽑는 이유는, 기간 일정이 월 경계를 넘을 때(`12.28 ~ 01.05`) 블록 월을 쓰면 틀리기 때문이다. 토큰이 더 정확하다.
2. **파싱 블록을 요청 연도로 필터**: 검증 중 `ssu.ac.kr`이 `?syear=`를 **조용히 무시하고 현재 연도(2026)를 반환**하는 것을 관찰했다. 만약 `?years=`도 언젠가 무시되면 `fetchCalendar(2018)`이 2026 데이터를 2018로 잘못 라벨링할 수 있다. 블록 id 연도가 요청 연도와 다르면 건너뛰어, 파라미터가 먹히지 않을 때 **잘못된 라벨 대신 빈 결과**가 나오게 했다(지금 고치는 버그와 같은 부류의 방어).
3. **기간은 시작일만 기존 `date`에 저장**: 소스에 `~` 범위가 있어 startDate/endDate 확장(보류됐던 ⑦)이 가능하지만, 그것은 공개 도구 응답 계약(DTO) 변경이므로 철칙2에 따라 **별도 결정으로 분리**한다. 이번 변경은 DTO 무변경으로 시작일만 `date`에 담는다.
4. **`category`는 항상 빈 문자열**: 실제 페이지는 일정마다 카테고리 라벨이 없다(모든 정보가 제목에 들어 있음). mock의 `"시험"`/`"방학"` 같은 분류는 진짜 데이터엔 없으므로, 분류기를 새로 만들지 않고 정직하게 빈 값으로 둔다.
5. **0건 시 mock으로 조용히 폴백하지 않음**: 빈 실제 결과는 경고 로그로 표면화하고, 안전망은 "배포 후 검증 + 설정 롤백"으로 둔다. 가짜 데이터로 빈 결과를 가리는 것은 지금 고치는 "조용히 틀림" 병 그 자체이기 때문이다.

### prod 활성화
기본값은 `mock`을 유지(개발·테스트가 라이브 사이트를 때리지 않도록)하고, prod만 `application-prod.yml`에서 `ssuai.connector.academic-calendar: real`로 오버라이드한다. **prod 설정 변경이므로 사용자 확인 후 적용**한다. 적용 후 배포 게이트로, prod에서 실제 2026 일정이 도구로 반환되는지 확인하고(로컬 curl로는 pod→ssu.ac.kr egress가 증명되지 않음) 문제 시 설정 롤백한다.

## 동작과 검증

1. `fetchCalendar(year)`가 `?years={year}` 페이지를 받아 `parseEvents(doc, year)`로 넘긴다.
2. `parseEvents`는 요청 연도와 일치하는 월 블록만 순회하며, 각 `li`에서 날짜 셀의 첫 `MM.DD` 토큰 + 블록 연도로 ISO 시작일을 만들고 제목과 함께 `AcademicCalendarEvent(date, event, "")`를 만든다.
3. 단위 테스트(네트워크 없음, 동결된 실제 2026 페이지 픽스처 `fixtures/campus/academic_calendar_2026.html`):
   - 스코프 셀렉터로 정확히 **63건** 파싱, 전부 `date`가 `2026-\d{2}-\d{2}`, `category` 빈 값.
   - 핀: `02.19 (목)` → `2026-02-19` & 제목에 "수강신청" 포함.
   - 기간 `01.05 ~ 01.28` → 시작일 `2026-01-05` 저장.
   - 연도 필터: 2026 블록만 있는 픽스처에 `parseEvents(doc, 2025)` → 빈 결과.
   - 토큰 우선: 블록 id 월(03) ≠ 토큰 월(05.10) → `2026-05-10`.
   - 빈 문서 → 빈 결과.
4. `find_academic_calendar_events`의 월 필터는 `date`의 `-MM-` 부분 일치에 의존하는데, 본 파서가 ISO `yyyy-MM-dd`를 생성하므로 호환된다.

## 포트폴리오 가치

"막혀서 검증 불가"로 보류돼 있던 항목을, 라이브 페이지를 단계적으로 조사해 **검증된 구조화 소스**로 바꾼 사례다. 특히 (a) 기존 커넥터가 한 번도 검증된 적 없었음(거짓 주석)을 404로 증명하고, (b) AJAX·헤드리스·OCR 가능성을 실제 페이지 조사로 하나씩 배제했으며, (c) `?syear=`가 조용히 무시되는 것을 보고 `?years=`에도 같은 방어(요청 연도 필터)를 넣은, "조용히 틀린 데이터"를 만들지 않는 설계 판단을 설명할 수 있다.

## 예상 면접 질문

1. 기존 커넥터의 "셀렉터를 확인했다"는 주석이 거짓임을 어떻게 증명했고, 왜 그게 단순 버그보다 위험한 신호인가요?
2. 연도는 블록 id에서, 월·일은 날짜 토큰에서 가져온 이유는 무엇이며 어떤 입력에서 차이가 드러나나요?
3. `?syear=`가 무시되는 것을 보고 파싱 단계에 요청 연도 필터를 넣었습니다. 이 방어가 막아주는 구체적 실패 시나리오는 무엇인가요?
