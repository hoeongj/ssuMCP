# ADR 0075 — 학사일정 기간 일정의 endDate 노출 (하위호환 DTO 확장)

| 항목 | 내용 |
|---|---|
| 날짜 | 2026-07-03 |
| 상태 | Accepted — 구현·머지 |
| 범위 | `AcademicCalendarEvent` DTO + `RealAcademicCalendarConnector` 범위 파싱 |
| 연관 문서 | ADR 0054(실소스 전환, 범위 파싱 보류 기록), ADR 0046(보류 ⑦), `docs/mcp-tools.md` |

---

## 배경

학사일정 소스 페이지의 상당수 행은 "01.05 (월) ~ 01.28 (수)" 같은 **기간 일정**인데, ADR 0054의 1차 구현은 시작일만 `date`로 노출하고 범위 정보를 버렸다(공개 DTO 계약 변경이라 별도 결정으로 보류 — ADR 0046 ⑦). 그 결과 클라이언트(웹 캘린더 위젯, LLM)는 "수강신청이 언제까지인지"를 알 수 없었고, 진행 중인 기간 일정이 "지난 일정"으로 오분류될 수 있었다.

## 검토한 대안

### `date`를 `startDate`/`endDate`로 하드 rename ❌
가장 깔끔한 명명이지만 기존 소비자(웹 클라이언트 타입, MCP 클라이언트의 도구 응답 파싱, 문서 예시)를 전부 동시에 깨는 브레이킹 체인지. 필드 하나 추가로 얻는 정보량이 같은데 굳이 rename 비용을 치를 이유가 없다.

### `startDate` 별칭 + `date` 유지 + `endDate` (3필드) ❌
과잉 — 같은 값이 두 필드로 나가면 어떤 게 canonical인지 오히려 혼란. LLM 소비자가 중복 필드에 서로 다른 의미를 상상하는 부작용도 있다.

### `date`(=시작일 의미 유지) + `endDate` nullable 추가 ✅ (채택)
기존 필드는 이름·의미(시작일) 그대로, 기간 일정에만 `endDate`(ISO, 포함)를 추가. 하루짜리는 `null`. 기존 소비자는 무변경으로 동작하고, 새 소비자는 `endDate != null`로 기간 여부를 판별한다.

## 핵심 결정

- **연말 경계 보정**: 소스 날짜는 연도가 없는 "MM.DD" 토큰이고 연도는 월 블록 id에서 온다. 범위의 종료 월이 시작 월보다 **작으면**(12월 블록의 "12.30 ~ 01.02") 종료 연도를 블록 연도 +1로 굴린다.
- **퇴화 범위 정규화**: "같은 날 ~ 같은 날" 행은 `endDate=null`(단일 일정)로 정규화 — 소비자가 범위/단일을 `endDate` 유무 하나로만 판별하게 유지.
- **mock도 동일 계약**: mock 커넥터 표본에 기간 일정(수강신청·시험 주간)을 포함시켜 프론트 개발/테스트가 실계약과 같은 모양을 보게 한다.

## 동작 방식

`RealAcademicCalendarConnector.parseDateRange()`가 날짜 셀에서 `MM.DD` 토큰을 최대 2개 추출: 첫 토큰=시작, 둘째 토큰(있으면)=종료. 각 토큰은 월/일 범위 검증 후 블록 연도와 결합해 ISO로 포맷. MCP 도구 `get_academic_calendar` 설명과 `docs/mcp-tools.md`에 필드 의미 명시. 웹 프론트(ssuAI)는 기간 일정을 "MM.DD ~ MM.DD"로 표시하고, '다가오는 일정' 필터가 `endDate`(없으면 `date`) 기준으로 진행 중 일정을 유지한다.

## 검증

`RealAcademicCalendarConnectorParseTests`: 기간 행 start+end 파싱, 단일 행 `endDate=null`, **연말 경계**(12월 블록 "12.30 ~ 01.02" → 2027-01-02), 기존 63-이벤트 픽스처 전수 파싱 회귀 유지. 컨트롤러/MCP 도구 테스트는 새 필드 포함 계약으로 갱신.

## 예상 면접 질문

1. **왜 rename 대신 nullable 필드 추가인가?** 공개 API의 필드 rename은 정보 이득 없이 모든 소비자를 깨는 비용만 있다. 시작일의 의미는 변하지 않았으므로 이름을 유지하고, 새 정보(종료일)만 추가하는 것이 하위호환 확장의 정석이다.
2. **연도 없는 날짜 소스의 함정은?** 연도를 문맥(월 블록)에서 복원해야 하는데, 범위가 연말을 넘으면 시작과 종료의 연도가 달라진다. "종료 월 < 시작 월 → +1년" 규칙과 이를 고정하는 테스트가 없으면 1월 종료일이 11개월 과거로 파싱된다.
3. **퇴화 범위를 왜 정규화했나?** 소비자 분기("기간인가?")를 `endDate != null` 하나로 유지하기 위해서다. "같은 날 범위"를 그대로 내보내면 소비자마다 동일-날짜 비교 로직이 중복된다.
