# ADR 0073 — 졸업사정표 ↔ 정책 근거 mismatch 경고 (보수적 정규식 cross-check, warn-only)

| 항목 | 내용 |
|---|---|
| 날짜 | 2026-07-02 |
| 상태 | Accepted — 구현·머지 |
| 범위 | MCP 도구 `evaluate_graduation_with_policy`의 응답에 `mismatchWarnings` 추가 |
| 연관 문서 | ADR 0020(학칙 하이브리드 RAG), ADR 0054(졸업요건 매핑), `docs/mcp-tools.md` |

## 배경

`evaluate_graduation_with_policy`는 학생의 u-SAINT 졸업사정표(rusaint 스크래핑 결과 — 총 이수학점·채플 등 요건별 required/completed)와 공식 학칙/졸업 근거(academic-policy 하이브리드 RAG 코퍼스의 조항 evidence)를 **나란히** 반환한다. 그러나 두 소스를 **교차 검증하지 않았다**: 사정표가 "채플 6회 필요"라고 하는데 인용된 학칙 조항이 다른 값을 말해도 그대로 통과했고, LLM 클라이언트가 둘 중 하나를 조용히 골라 학생에게 단정할 위험이 있었다. 이 불일치를 명시적 신호로 표면화해 "학교 시스템과 학칙 문서가 서로 다른 값을 말하니 학과 사무실에 확인하라"고 안내하게 하는 것이 목표다. **두 데이터 소스는 수정하지 않는 데이터 품질 가드레일**이다.

## 검토한 대안

### LLM-judge 비교 ❌
evidence 스니펫과 사정표를 LLM에 주고 "일치하냐" 묻는 방식. 기각 — 비결정적(같은 입력에 다른 판정)·비용·지연이 붙고, 데이터 품질 신호는 재현 가능해야 한다. 경고가 매번 달라지면 신뢰할 수 없다.

### fail-closed 차단(불일치 시 응답 거부/막기) ❌
기각 — 정책 코퍼스는 **참고 컨텍스트이지 권위 소스가 아니다**. 학과별 교육과정·입학연도 경과조치 때문에 학칙 일반 조항의 숫자가 특정 학생에게 안 맞는 게 정상일 수 있다. 불일치를 오류로 취급해 기능을 막으면 오히려 잘못된 권위 부여다. 경고만 띄우고 판단은 사람에게 넘긴다.

### 보수적 정규식 cross-check, warn-only ✅ (채택)
evidence가 **이미 반환한** 텍스트(heading+snippet)에서 요건 키워드에 앵커된 숫자를 정규식으로 추출해 사정표 값과 비교한다. 추가 RAG 호출 없음. **한 evidence 청크에서 그 요건의 숫자 후보가 정확히 하나로 확정될 때만** 주장을 세우고, 사정표와 다르면 경고를 추가한다.

## 핵심 결정: 정직성·저 오탐

- **대상 요건 2종만**: 숫자 의미가 모호하지 않은 **총 이수학점**(100–199, `졸업/이수/취득`+`학점` 앵커)과 **채플**(1–8, `채플`+`학기/회` 앵커). 전공/교양 등은 학과 의존이 커 숫자 단정이 위험하므로 제외.
- **단일 후보만 신뢰**: `singleNumber()`는 청크에서 서로 다른 값이 2개 이상 잡히면 **주장을 포기**(모호 → 무경고). 예: "졸업학점은 130학점이며 일부 학과는 140학점" → 두 값 → 침묵.
- **seed 코퍼스에는 하드 넘버가 없다**: seed 코퍼스는 의도적으로 구체 숫자를 담지 않는다(요건이 학과·입학연도별로 달라서). 따라서 이 검출기는 **seed-only evidence에는 자연히 침묵**하고, `live=true`가 실제 숫자를 명시한 공식 조항을 가져올 때 주로 발화한다. 이것이 오탐을 0에 가깝게 만드는 구조적 이유다.
- **부동소수 비교**: 사정표 `graduationPoints`(float)와 정책 int를 `abs(diff) < 0.5`로 비교해 133.0 vs 133 같은 노이즈를 무경고 처리.

## 동작 방식

`GraduationPolicyMismatchDetector.detect(status, brief)`가 evidence마다 heading+snippet을 이어 붙여 두 패턴을 돌리고, 각 요건에서 단일 숫자가 확정되고 사정표와 0.5 이상 벌어지면 `GraduationPolicyMismatchWarning{requirementKey, assessmentValue(float), policyValue(int), evidenceTitle, evidenceHeading, message}`를 만든다. 도구는 이 리스트를 `GraduationPolicyEvaluationResponse.mismatchWarnings`(항상 존재, 기본 빈 리스트)로 실어 반환한다.

응답 예시(불일치 발생 시):

```json
"mismatchWarnings": [
  {
    "requirementKey": "CHAPEL_SEMESTERS",
    "assessmentValue": 6.0,
    "policyValue": 4,
    "evidenceTitle": "학칙시행세칙(학사과정)",
    "evidenceHeading": "채플",
    "message": "채플 이수 기준이 u-SAINT 졸업사정표(6회)와 학칙 근거(4회)에서 다릅니다. 입학연도별 채플 이수 기준을 학과 사무실 또는 학사팀에 확인하세요."
  }
]
```

## 한계

- 한글 조항 표현에 튜닝된 정규식이라 표기가 크게 다른 원문(표/이미지·비정형)에서는 추출이 안 될 수 있다(→ 무경고, 안전한 실패).
- 요건 2종만 커버. 전공·교양·인증 등은 학과 의존이 커 의도적으로 뺐다.
- 정책 숫자가 특정 학생에게 안 맞는 게 정상인 경우도 경고가 뜬다 — 그래서 메시지가 "틀렸다"가 아니라 "다르니 확인하라"다.

## 검증

`GraduationPolicyMismatchDetectorTests` 9건: 총학점/채플 불일치→경고(양쪽 값+evidence 참조 검증), 일치→무경고, 모호(서로 다른 두 숫자)→무주장, seed-스타일 무숫자→무경고, evidence 없음→무경고, 총학점+채플 동시 경고, null 안전. `AcademicPolicyMcpToolsTests`는 응답 조립 경로를 커버(빈 evidence→빈 warnings).

## 예상 면접 질문

1. **왜 LLM에게 비교를 맡기지 않았나?** 데이터 품질 신호는 재현 가능·저비용·결정적이어야 한다. LLM 판정은 비결정적이라 같은 학생에 경고가 떴다 안 떴다 하면 신뢰를 잃는다. 숫자 대조는 규칙으로 충분하고, 애매하면 침묵하는 편이 안전하다.
2. **오탐을 어떻게 0에 가깝게 만들었나?** ① 요건을 숫자 의미가 명확한 2종으로 한정 ② 청크당 단일 숫자 후보로 확정될 때만 주장(모호하면 포기) ③ seed 코퍼스가 하드 넘버를 안 담는 특성상 자연 침묵 — "확신 없으면 말 안 한다"를 세 겹으로 강제했다.
3. **불일치를 왜 차단이 아니라 경고로 했나?** 정책 코퍼스는 권위 소스가 아니라 참고 컨텍스트다. 학과·입학연도별 경과조치 때문에 일반 조항 숫자가 특정 학생과 다른 게 정상일 수 있어, 오류로 막으면 잘못된 권위 부여가 된다. 판단은 사람(학과 사무실)에게 넘기고 시스템은 신호만 준다.
