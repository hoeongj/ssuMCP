# ADR 0050 - 좌석 대기 자동 예약의 기존 이용 내역 멱등성

- **상태**: Accepted - 2026-06-20 구현
- **날짜**: 2026-06-20
- **범위**: `LibraryReservationWorker.prepare()`의 비즉시 좌석 대기 처리

## 배경

좌석 대기 worker는 조건에 맞는 빈 좌석을 찾으면 별도 확인 없이 자동 예약한다. 그러나 사용자가 이미 다른 경로에서 활성 좌석 이용 내역(charge)을 얻은 뒤 오래된 "아무 좌석" 대기가 남아 있으면, worker가 다른 좌석을 다시 예약해 이중 예약을 시도할 수 있었다. 기존의 좌석별 분산 락과 Postgres 상태 전이는 같은 좌석이나 같은 intent의 중복 실행은 막지만, 한 사용자가 서로 다른 좌석을 보유하는 문제는 막지 못한다.

## 결정

`isImmediateReservation()`이 `false`인 비즉시 대기에만 좌석 탐색 전 `getCurrentCharge(token)`을 호출한다. 활성 charge가 있으면 기존 좌석 ID와 charge 요약을 기록해 intent를 성공 처리하고, 좌석 탐색과 `reserve()`는 실행하지 않는다.

즉시/목표 좌석 예약에는 이 검사를 적용하지 않는다. 목표 좌석 예약은 사용자가 의도한 좌석 업그레이드나 이동일 수 있으므로 기존 좌석 보유만으로 종료하면 명시적 행동을 깨뜨린다.

## 대안과 기각 이유

- **모든 예약 intent를 건너뛴다**: 구현은 단순하지만 목표 좌석으로 이동하려는 즉시 예약까지 막아 업그레이드 동작을 회귀시키므로 기각했다.
- **기존 charge를 확인하지 않는다**: 추가 조회가 없지만 오래된 아무 좌석 대기가 다른 좌석을 자동 예약하는 이중 예약 위험이 남으므로 기각했다.

## 동작 방식

1. session token이 없으면 기존과 같이 인증 실패 처리한다.
2. 즉시 예약이면 기존 경로대로 목표 좌석을 예약한다.
3. 비즉시 대기이면 현재 charge를 조회한다.
4. charge가 있으면 `SUCCEEDED`로 종료하고 기존 좌석 ID 및 `successMessage()` 형식의 요약을 남긴다.
5. charge가 없으면 기존 좌석 선택과 예약 흐름을 그대로 수행한다.
6. 조회 중 인증 예외는 인증 실패로 종료한다. 그 밖의 일시적 런타임 예외는 경고를 남기고 기존 좌석 탐색을 계속해, 확인 API 장애가 정상 대기를 영구 차단하지 않게 한다.

테스트는 기존 charge가 있는 비즉시 대기의 조기 성공, charge가 없는 정상 예약, 기존 charge가 있어도 즉시/목표 예약은 계속되는 세 경계를 네트워크 없이 검증한다.

## 예상 면접 질문

1. 좌석별 분산 락이 있는데도 사용자별 기존 charge 검사가 필요한 이유는 무엇인가?
2. 즉시/목표 좌석 예약에는 같은 멱등성 검사를 적용하지 않은 이유는 무엇인가?
3. 기존 charge 조회 실패 시 hard-fail 대신 좌석 탐색을 계속하도록 선택한 가용성·정합성 trade-off는 무엇인가?
